## moe训推不一致
moe基础

一、 Top-K 路由机制 (Top-K Routing Mechanism)这是 MoE 模型最核心的组件，决定了当前输入的 Token 应该交给哪些“专家”来处理。你需要掌握的核心知识点：门控网络 (Gating Network / Router)： 它的本质是一个简单的线性分类器。对于输入的 Token 向量 $x$，通过一个权重矩阵 $W_g$ 映射到专家数量的维度上，然后经过 Softmax 计算出每个专家的概率（打分）。$$G(x) = \text{Softmax}(W_g \cdot x)$$Top-K 选择与稀疏激活： 为了节省算力，模型不会让所有专家都参与计算，而是只选取概率最高的 K 个专家（通常 K=2 或 K=4）。加权求和输出： 选出的 K 个专家分别对 Token 进行前向传播计算，然后将它们的输出结果与 Router 给出的概率分数进行加权求和，得到最终的输出 $y$。$$y = \sum_{i \in \text{TopK}} G(x)_i \cdot E_i(x)$$Qwen 架构的特殊性（加分项）： 你在项目中使用了 Qwen3 的 MoE 版本。Qwen 系列（如 Qwen1.5/2-MoE）引入了 共享专家 (Shared Experts) 机制。即除了按 Top-K 路由的专家外，还有几个专家是所有 Token 都必须经过的（用于提取通用语义特征），这与传统的纯路由 MoE（如 Mixtral）有所不同，面试时点出这一点会非常亮眼。二、 专家负载均衡损失 (Load Balancing Loss)如果没有外力干预，门控网络很容易“偷懒”——产生马太效应 (Matthew Effect)。即它会发现某几个专家特别好用，就把所有 Token 都扔给它们，导致这几个专家被“撑死”（计算过载），而其他专家被“饿死”（参数得不到更新）。你需要掌握的核心知识点：辅助损失函数 (Auxiliary Loss) 的构成： 为了强迫模型将 Token 均匀分配给所有专家，在标准的交叉熵 Loss 之外，会人为加上一个负载均衡 Loss。它通常由两部分组成：$f_i$ (Fraction of tokens)： 在一个 Batch 中，实际被分配给第 $i$ 个专家的 Token 比例。$P_i$ (Average probability)： 在一个 Batch 中，门控网络分配给第 $i$ 个专家的平均预测概率。计算公式： 假设有 $N$ 个专家，$\alpha$ 为权重系数。公式设计的巧妙之处在于，只有当 $f_i$ 和 $P_i$ 都趋近于均匀分布（即 $\frac{1}{N}$）时，这两个向量的内积（即 Loss 值）才最小。$$L_{bal} = \alpha \cdot N \sum_{i=1}^N f_i \cdot P_i$$工程落地点： 在训练阶段，如果某个专家的队列满了（Capacity Factor），多出来的 Token 会被直接丢弃（Token Dropping）或者通过残差连接跳过，这会严重影响训练效果，这也是为什么必须优化负载均衡的原因。三、 SFT 阶段的专家坍塌 (Expert Collapse) 及缓解策略预训练阶段数据汪洋大海，各个领域的数据都有，专家们能做到“各司其职”。但在 SFT 阶段，数据通常高度垂直且规模较小（例如你构建的 1 万条金融意图分发数据），这极易引发问题。你需要掌握的核心知识点：专家坍塌的现象： 由于 SFT 数据分布单一，Router 会迅速收敛，把几乎所有该领域的 Token 都固定路由给某 1 到 2 个专家。这导致 MoE 退化成了普通的小参数 Dense 模型，丧失了海量参数带来的优势，并且容易造成严重的过拟合。缓解策略 (Mitigation Strategies)：冻结路由层 (Freeze Router)： 这是工业界做 MoE 微调最简单也最常用的方法。在 SFT 时，直接冻结门控网络 $W_g$ 的参数，只微调专家的参数。这样模型依然保留了预训练时学到的路由分配逻辑，避免了 Token 扎堆。调整负载均衡权重： 在 SFT 阶段适当调大 $L_{bal}$ 中的 $\alpha$ 系数，通过强力的正则化惩罚，逼迫 Router 将垂直领域的数据也尽量分散给不同专家处理（但可能会轻微损伤模型在该领域的极致性能）。专家级 LoRA (MoE-LoRA / MoRA)： 不去全量微调专家，而是针对每个专家挂载独立的 LoRA 模块，或者在所有专家外部挂载一个全局共享的 LoRA，让 LoRA 去拟合 SFT 的垂直分布，而保持底层专家的多样性不被破坏。
